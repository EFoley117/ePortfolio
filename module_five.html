Unit 10:API and Case Analysis

Unit 10 will analyse and evaluate APIs (Application Programming Interface) and how APIs facilitate data parsing and inter process communication. We will examine the security requirements for enabling the robust functionality of APIs as well as the challenges and issues involved.

Summarising this modules studies:

  • Evaluate APIs and its use in data parsing and inter process communication.
  • Examine the security requirements for ensuring the resilience of the functionality of APIs.
  • Examine the challenges and issues associated with the implementation of APIs.

API Security Requirements

evaluate the security requirements of an API of your choice and write a brief security requirements specification which mitigates against any risks associated with the API for enabling data sharing, scraping and connectivity between a program code written in Python and any of the following file formats/management systems (XML, JSON and SQL).

Application programming interface (API) is a set a coded routines which both accepts requests and formulates and sends responses to neighbouring programmes. An example of an API, is google maps API this meets the definition as it is a series of programming tools which permits developers to merge functionalities within google maps into their own applications, websites and services thus allowing integration of google maps data.

One of the key security concerns with API technology surrounds authorisation controls, unpermitted access from external parties with malicious intent could result in unwarranted data access or malicious programme modification.1 This concept refers to Insecure direct object references, whereby users can manipulate data parameters, for example in google maps this could involve altering the geolocation of a business address. This security vulnerability can be stopped at the Data validation part of the data wrangling process, whereby data inputted from injection attacks e.g. SQL injection or XSS can be identified and prevented.3

Another form of security risk associated with API include systems having a lack of rate limiting and Throttling mechanisms in place, this can make the API susceptible to Denial-of-Service attaches, whereby the system receives a significant volume of data which will invariably lead to excessive usage age and performance. Throttling software can effectively prevent this issue from occurring.

API software is also vulnerable to unauthorised access via a user’s browser, which can result in users making unauthorised requests resulting in unpermitted access or data leakage. A common source of such breach is misconfiguration in the Cross-Origin Resource sharing process. It is important for Google maps ensure that these pathways permit only developers on their end of the system have access to such pathways.2

A further facet of the data management pipeline that can leave Google Maps API vulnerable to security attacks includes the volume of data that is collected and stored. The mass assignment of data without proper validation protocols leaves the software vulnerable to manipulation attacks, secondary to the sheer volume of data being collected and collated – this issue can also be addressed at the data wrangling stage of the management pipeline.4

The Key phase in googles maps API data management pipeline surrounds the use of effective data validation methods. There are several methods which can be deployed to optimise the validation:

  • • Input validation – this essentially ensures that all incoming data adheres to the correct data format, data type and range. This can be further checked using schema validation, whereby the structure of the expected incoming data can be clearly
  • • The developers are able to set parametrised queries, this ensures that user inputs are not treated as code which could affect the functionality of the API.
  • • Content type validation, whereby the header of expected requests matches the expected data format e.g. JSON or XML to prevent potential parsing issues.1

It is important for Google API to conduct regular reviews of their data validation pathways in response to new and emerging threats and continue to perform security assessments such as penetrating testing to identify potential weakness’ in the process.
